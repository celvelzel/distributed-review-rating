## 1. 项目概述

### 1.1 项目背景

本项目是 COMP5434 Big Data Computing 课程的 Kaggle 竞赛项目，任务是**预测电商评论的评分（1-5星）**。训练集包含约 300 万条评论，测试集包含 1 万条评论。

### 1.2 项目目标

- **主要目标**: 使用分布式数据处理框架构建准确的评分预测系统
- **评估指标**: RMSE（均方根误差），数值越低越好
- **当前最佳成绩**: **Kaggle RMSE 0.59770**（从初始基线 0.801 改进 25.4%）

### 1.3 数据规模

| 数据集 | 行数 | 说明 |
|--------|------|------|
| train.csv | 2,949,249 | 训练数据（含评分标签） |
| test.csv | 10,000 | 测试数据（需预测评分） |
| prodInfo.csv | 213,593 | 产品元数据 |

### 1.4 评分分布

| 评分 | 数量 | 占比 |
|------|------|------|
| 5星 | 1,637,304 | 55.52% |
| 4星 | 438,166 | 14.86% |
| 3星 | 290,240 | 9.84% |
| 2星 | 242,311 | 8.22% |
| 1星 | 341,228 | 11.57% |

**关键发现**: 评分分布严重不平衡，5星评论占比超过一半。

## 2. 优化进程

### 2.1 优化时间线总览

| 日期 | 最佳 RMSE | 关键技术 | 改进幅度 |
|------|-----------|----------|----------|
| 06-06 | 0.801 | TF-IDF + LightGBM 基线 | — |
| 06-06 | 0.790 | LightGBM 正则化 | +1.4% |
| 06-07 | 0.699 | 多样性集成（LGB+XGB+MLP） | +11.5% |
| 06-13 | 0.664 | Ridge Stacking（6个模型） | +5.1% |
| 06-13 | 0.638 | DeBERTa LoRA 微调 | +3.8% |
| 06-14 | **0.617** | 方差扩展（VE）突破 | +3.3% |
| 06-20 | **0.598** | VE比例优化（60/40） | +3.2% |

**总改进**: 从 0.801 到 0.598，**改进幅度 25.4%**

### 2.2 第一阶段：基线建立（06-05 至 06-06）

#### 目标
建立可复现的基线模型，验证数据处理流程。

#### 方法
- 使用 TF-IDF（词频-逆文档频率）提取文本特征
- 使用 LightGBM（梯度提升树）训练模型

#### 结果
- **OOF RMSE**: 1.176（本地交叉验证）
- **Kaggle RMSE**: 0.801

#### 关键发现
- 文本特征（TF-IDF）是最有效的特征
- 简单模型即可获得合理基线

### 2.3 第二阶段：目标泄漏发现与修复（06-06 至 06-07）

#### 问题
添加统计特征（用户平均评分、产品平均评分）后，本地 RMSE 降至 0.55，但 Kaggle 分数反而升至 1.50。

#### 原因分析
- **目标泄漏**: 统计特征直接使用了要预测的评分
- 即使使用 K-Fold 编码，模型仍能记忆训练集模式
- **教训**: 如果 OOF RMSE < 0.10，几乎必定是泄漏

#### 解决方案
- 使用 K-Fold 目标编码，确保每个样本的统计特征仅来自其他折的数据
- 开发 37 个单元测试验证无泄漏

### 2.4 第三阶段：模型多样性探索（06-07 至 06-10）

#### 尝试的方法

| 方法 | Kaggle RMSE | 结果 |
|------|-------------|------|
| LightGBM 正则化 | 0.790 | ✅ 改进 |
| XGBoost | 0.802 | ❌ 稍差 |
| MLP（BERT嵌入） | ~0.70 | ✅ 显著改进 |
| 等权重集成 | 0.807 | ❌ 更差 |
| 加权集成（网格搜索） | 0.699 | ✅ 最佳 |

#### 关键发现
- **MLP 主导集成**: 尽管 MLP 单模型 OOF 最差（1.131），但在集成中获得 86% 权重
- **原因**: MLP 提供了与树模型正交的信号（多样性）
- **教训**: 集成多样性 > 单模型性能

### 2.5 第四阶段：Transformer 微调（06-10 至 06-14）

#### DeBERTa-v3-base LoRA 微调

**配置**:
- 模型: microsoft/deberta-v3-base（8600万参数）
- 训练数据: 100万样本（从300万中随机采样）
- 训练配置: 5折 × 5 epoch
- LoRA: r=16, alpha=32, 仅训练 0.32% 参数（589K/184M）
- 损失函数: CORAL 序数损失 + R-Drop 正则化

**结果**:
- **OOF RMSE**: 1.117
- **Kaggle RMSE**: 0.638

#### 方差扩展（VE）突破

**问题**: DeBERTa 预测严重压缩
- 预测标准差: 0.825
- 真实标签标准差: 1.422
- 比率: 仅捕获 58% 的真实方差

**解决方案**:
```python
# 方差扩展公式
scale = target_std / pred_std  # = 1.422 / 0.825 = 1.72
pred_calibrated = (pred - pred_mean) * scale + target_mean
```

**效果**: Kaggle RMSE 从 0.638 提升至 0.617（**单技术改进 3.3%**）

## 3. 分布式设计

### 3.1 分布式框架选择

**框架**: Apache Spark（PySpark 3.4.1）  
**运行模式**: local[*]（伪分布式，使用所有 CPU 核心）  
**满足课程要求**: 项目规范要求至少使用一个分布式处理框架

### 3.2 Spark 配置

| 配置项 | 值 | 说明 |
|--------|-----|------|
| master | local[*] | 使用所有可用 CPU 核心 |
| driver.memory | 4g（ETL）/ 20g（TF-IDF） | 根据任务调整 |
| shuffle.partitions | 200 | 分布式洗牌并行度 |
| broadcastTimeout | 600s | 广播超时时间 |

### 3.3 ETL 流水线架构

```
原始 CSV 文件 (data/)
    │
    ▼
┌─────────────────────────────────────────────┐
│  阶段 1: ETL (PySpark)                       │
│  1. 加载 CSV → Spark DataFrame              │
│  2. 清洗文本（HTML、URL、特殊字符）          │
│  3. 填充缺失值（标题→'unknown'，票数→0）    │
│  4. 广播连接: 训练集 ∞ 产品信息              │
│  5. 提取时间特征（年、月、星期、小时）       │
│  6. 持久化 → Parquet                        │
└─────────────────────────────────────────────┘
    │
    ▼ artifacts/etl/{train,test,prodinfo}.parquet
    │
┌─────────────────────────────────────────────┐
│  阶段 2: 特征工程                            │
│  ┌──────────────────┐  ┌──────────────────┐ │
│  │ 统计特征          │  │ 文本特征         │ │
│  │ • 用户统计        │  │ • TF-IDF 50K     │ │
│  │ • 产品统计        │  │ • 字符 TF-IDF    │ │
│  │ • 类别统计        │  │ • SVD 降维       │ │
│  │ • 价格特征        │  │ • DeBERTa 嵌入   │ │
│  │ • 时间特征        │  │ • LightGCN 嵌入  │ │
│  └──────────────────┘  └──────────────────┘ │
└─────────────────────────────────────────────┘
    │
    ▼ artifacts/features/*.parquet, *.npz, *.npy
    │
┌─────────────────────────────────────────────┐
│  阶段 3: 模型训练与集成                      │
│  • 9 个基础模型训练                          │

│  • DeBERTa LoRA 微调                        │
│  • VE 校准 + 加权混合                       │
└─────────────────────────────────────────────┘
    │
    ▼ output/submission-*.csv
```

### 3.4 关键分布式操作

#### 3.4.1 广播连接（Broadcast Join）

```python
# 将小表（产品信息 213K 行）广播到所有工作节点
from pyspark.sql.functions import broadcast
df.join(broadcast(df_prodinfo), on="parent_prod_id", how="left")
```

**优势**: 避免昂贵的洗牌连接，将小表复制到每个节点。

#### 3.4.2 分布式 TF-IDF

使用 PySpark ML Pipeline 实现分布式 TF-IDF：
```
RegexTokenizer → NGram(2,3) → HashingTF(50K) → IDF → VectorAssembler
```

#### 3.4.3 分布式聚合

```python
# 用户级别统计（分布式 groupBy）
user_stats = df.groupBy("user_id").agg(
    F.avg("rating").alias("avg_rating"),
    F.count("rating").alias("review_count"),
    F.stddev_pop("rating").alias("rating_std")
)
```

### 3.5 内存管理策略

| 策略 | 说明 |
|------|------|
| 广播连接 | 小表广播避免洗牌 |
| 分块 Parquet 写入 | 每次写入 50 万行，避免 OOM |
| 增量批训练 | LightGBM 逐行组读取，峰值内存 ~14GB |
| 子采样交叉验证 | 在 50 万-100 万行子集上运行 CV |
| 内存映射嵌入 | 使用 mmap_mode="r" 加载嵌入 |

### 3.6 数据格式选择

| 格式 | 用途 | 优势 |
|------|------|------|
| Parquet | 中间数据存储 | 列式存储，10-100x 读取加速 |
| Sparse .npz | TF-IDF 特征 | 避免稠密存储 50K 维向量 |
| Dense .npy | 嵌入特征 | BERT 768维、LightGCN 64维 |

## 5. 项目亮点

### 5.1 方差扩展（VE）突破

**问题**: DeBERTa 预测严重压缩，仅捕获 58% 的真实方差。

**创新**: 使用简单的线性缩放恢复预测分布。

**效果**: 单技术改进 3.3%，是项目中最大的单项技术突破。

**意义**: 证明了后处理校准的重要性，即使模型预测排名准确，也需要校准预测值的分布。

### 5.2 训练配置 > 数据量

**发现**: 100万数据 + 5折×5epoch 配置优于 300万数据 + 3折×3epoch。

**原因**: 
- 100万数据的 scheduler 总步数较少（3125 vs 23438）
- epoch1 即可完成完整的余弦退火
- 更多折数提供更好的 OOF 估计

**教训**: 不要盲目增加数据量，训练配置同样重要。

### 5.3 集成多样性 > 单模型性能

**发现**: MLP 单模型 OOF 最差（1.131），但在集成中获得 86% 权重。

**原因**: MLP 提供了与树模型正交的信号（多样性）。

**Ridge 系数分析**:
- ensemble_diverse: +0.745（元集成，最强贡献）
- lgb_graph_safe: +0.404（图特征，提供多样性）
- mlp: +0.145（DeBERTa 嵌入）

**教训**: 集成中应优先考虑模型多样性，而非单模型性能。

### 5.4 系统性目标泄漏检测

**问题**: 统计特征（用户平均评分、产品平均评分）导致目标泄漏。

**解决方案**: 
- 开发 37 个单元测试验证无泄漏
- 使用黄金标准方法：手动计算期望值并与实际输出比较
- 建立规则：OOF RMSE < 0.10 几乎必定是泄漏

### 5.5 OOF 与 Kaggle 相关性分析

**发现**: OOF RMSE 与 Kaggle RMSE 存在中等强度正相关（r≈0.75），但存在系统性偏移（~0.43）。

**可靠性分类**:
- ✅ 高: 同架构同数据超参数调优
- ⚠️ 中: 不同模型类型
- ❌ 低: 集成权重优化、VE 比例调整
- ❌ 无: 后处理（VE、blending）

**实用建议**: 用 OOF 粗筛，用 Kaggle 精选。

### 5.6 资源约束下的优化

**约束**: HPC 内存限制 15GB，DeBERTa-v3-base 需要 ~12GB。

**解决方案**:
- 梯度检查点: GPU 内存从 4.2GB 降至 0.9GB
- 数据子采样: 使用 100万而非 300万
- 较小批量: 16 而非 32
- GPU 交换策略: 在不同模型间切换 GPU 使用

### 5.7 完整的实验追踪

**实践**: 
- 每次实验记录详细日志
- 使用 JSON 文件存储结构化指标
- 维护完整的历史提交记录（24个提交）
- 定期生成状态报告

**价值**: 避免重复错误，快速定位问题，支持决策。

#### 6.1.3 图特征效果有限

**问题**: 
- 图太稀疏（3M 边 / 2M 节点，平均度 ~3）
- SVD 仅解释 7% 方差
- LightGCN 嵌入 OOF = 1.4186（几乎等于 baseline）

**结论**: 当前图特征对最终模型贡献有限。

#### 6.1.4 OOF 与 Kaggle 不一致

**问题**: 
- OOF RMSE 与 Kaggle RMSE 存在 ~0.43 的系统性偏移
- 相关系数 r≈0.75，不是完美预测器
- 集成权重优化、VE 比例调整的 OOF 改善无法可靠转化为 Kaggle 改善

**影响**: 必须频繁提交 Kaggle 验证实际效果。

#### 6.1.5 每日提交限制

**约束**: Kaggle 每天允许约 10 次提交。

**影响**: 限制了实验迭代速度，必须优先提交最有希望的变体。
